# SYSTOLAB API

The SYSTOLAB API is a self-owned deterministic intelligence API. It does not call Google PageSpeed, paid SEO APIs, analytics APIs, AI APIs, or any external scoring API.

## Production Security Controls

Phase 1 hardening enforces production startup validation and scanner abuse protection.

Production startup requires real environment values for `MONGODB_URI`, `CLIENT_ORIGIN`, `SYSTOLAB_AUTH_JWT_SECRET`, `SYSTOLAB_INTERNAL_ADMIN_KEY`, `SYSTOLAB_OWNER_ADMIN_KEY`, `SYSTOLAB_MANAGER_ADMIN_KEY`, `SYSTOLAB_GOOGLE_CLIENT_ID`, and `SYSTOLAB_GOOGLE_JWKS_JSON`. Production rejects placeholder, weak, or development-style secrets, rejects invalid MongoDB URI formats, rejects `SYSTOLAB_MEMORY_STORE=true`, and disables sandbox Google credentials.

Scanner URLs are validated before scan creation and before every crawler request. SYSTOLAB allows only HTTP and HTTPS, removes fragments, rejects embedded credentials, blocks localhost and internal hostnames, blocks private/local/metadata/reserved IPv4 and IPv6 addresses, validates every DNS result, pins the validated DNS result during the request, and revalidates every redirect target. HTTPS-to-HTTP redirects are rejected.

Abuse controls are layered:

- Global API limit through `SYSTOLAB_GLOBAL_RATE_LIMIT_PER_MINUTE`.
- Authentication route limit through `SYSTOLAB_AUTH_RATE_LIMIT_PER_MINUTE`, plus OTP/login retry locks in the auth service.
- Scan creation limit through `SYSTOLAB_SCAN_RATE_LIMIT_PER_MINUTE`; scan job status polling uses `SYSTOLAB_SCAN_STATUS_RATE_LIMIT_PER_MINUTE`.
- Public API limit through `SYSTOLAB_PUBLIC_API_RATE_LIMIT_PER_MINUTE`, plus daily API quota enforcement through `SYSTOLAB_PUBLIC_API_DAILY_QUOTA`.

## Internal UI API

`POST /api/scans`

```json
{
  "targetUrl": "https://example.com",
  "mode": "full_audit",
  "includeSeo": false,
  "gbpUrl": "https://www.google.com/maps/place/example",
  "competitorUrls": ["https://competitor.example"],
  "monthlyLeadVolume": 100
}
```

`GET /api/reports/:snapshotId`

Returns the full customer report `ReportSnapshot`, including decision, insight, proof, GBP, competitor, recommendation, evidence, monitoring, outcome validation, and report governance sections.

`GET /api/reports/:snapshotId/decision`

Returns the compact Customer Intelligence Plane output generated by the Decision Compression Layer for API consumers that need a small decision object. This AICE Decision Object is limited to:

- `scan_id`
- `target`
- `risk_level`
- `time_sensitivity`
- `evidence_summary`
- `coverage_score`
- `confidence_score`
- `revenue_impact_range`
- `if_not_fixed_outcome`
- `recommended_action_window`
- `final_recommendation`
- `access_restriction_detected`
- `assessment_limitation`
- `impact`
- `evidence_heatmap_summary`

The Decision Compression Layer rejects customer output that contains raw evidence objects, DOM snapshots, HTTP snippets, validation traces, telemetry, evidence IDs, score trace factors, internal governance records, internal architecture state, execution traces, parser outputs, recovery logs, compatibility intelligence, failure memory, system diagnostics, or intermediate scoring metrics.

When current-scan evidence is incomplete, blocked, inaccessible, or insufficient, customer output automatically downgrades confidence, can return `risk_level: "UNKNOWN"`, sets unsupported value ranges to zero, and communicates limitations without exposing internal mechanics.

The full browser report and full PDF intentionally include complete diagnostic/reporting information. Server credentials, auth tokens, environment secrets, and database secrets are never report fields and are never exposed.

`GET /api/internal/platform/reports/:snapshotId/full`

Returns the complete Internal Intelligence Plane `ReportSnapshot` for authorized SYSTOLAB administrators. This route requires an admin bearer token and is used by the internal full report view at `/internal/reports/:snapshotId`.

`GET /api/reports/:snapshotId/pdf`

Returns the full generated PDF report.

`GET /api/coverage`

Returns the requirement coverage registry.

Internal intelligence artifacts are persisted when MongoDB is connected:

- `Workspace`: tenant and target context for a business.
- `ScanHistory`: immutable scan history points for a workspace.
- `IntelligenceEvent`: standardized event envelopes across SYSTOLAB layers.
- `RecommendationRecord`: action-ready recommendation records.
- `ChangeRecord`: lightweight delta records linked to snapshots and recommendations.
- `EvidenceRecord`: before/after evidence rows with confidence and lineage.
- `OutcomeValidationRecord`: recommendation outcome validation proof.
- `AlertRecord`: dashboard alert records.
- `MonitoringSchedule`: recurring scan cadence and next-run state.
- `NotificationOutbox`: dashboard and email-simulated alert delivery jobs.
- `EditEvent`: first-party dashboard/edit/funnel events.
- `CompetitorRelationshipRecord`: CRG-ready business-to-competitor relationship observations with industry, geography, segment, OSS, trust, conversion, revenue opportunity, and threat signals.
- `UserMemory`: graph-ready memory entries for future operational memory features.

## Intelligence API

All routes are self-owned and use SYSTOLAB persistence only.

- `GET /api/intelligence/evidence/:snapshotId`: returns persisted Evidence Database rows.
- `GET /api/intelligence/outcomes/:snapshotId`: returns persisted recommendation outcome validations.
- `GET /api/intelligence/alerts`: returns dashboard alert records. Optional `targetUrl` query is supported.
- `GET /api/intelligence/notifications`: returns self-owned notification outbox jobs for dashboard and email-simulated alert channels.
- `GET /api/intelligence/monitoring/schedules`: lists monitoring schedules.
- `POST /api/intelligence/monitoring/schedules`: creates or updates a daily, weekly, or monthly monitoring schedule.
- `POST /api/intelligence/monitoring/run-due`: manually executes due monitoring schedules through the same worker cycle used by the server.
- `POST /api/intelligence/edit/events`: records first-party dashboard/edit/funnel events for Edit Intelligence.

The API server starts the monitoring worker automatically when `SYSTOLAB_MONITORING_WORKER_ENABLED` is not `false`. Worker controls:

- `SYSTOLAB_MONITORING_WORKER_ENABLED`
- `SYSTOLAB_MONITORING_WORKER_INTERVAL_MS`
- `SYSTOLAB_MONITORING_WORKER_BATCH_SIZE`

Alert delivery is self-contained. `email_simulated` creates an internal `NotificationOutbox` job marked `delivered_simulated`; it does not call SMTP, SMS, Google, or any third-party messaging API.

## Internal IIRE API

The Internal Intelligence Reporting Engine is never customer-facing. Every route below requires:

```http
x-systolab-internal-key: your_internal_admin_key
```

`SYSTOLAB_INTERNAL_ADMIN_KEY` controls access.

- `GET /api/internal/iire/reports`: list generated internal intelligence reports.
- `POST /api/internal/iire/reports`: generate a daily, weekly, monthly, quarterly, annual, custom, or event-triggered internal report.
- `GET /api/internal/iire/reports/:reportId`: retrieve one internal report.
- `GET /api/internal/iire/reports/:reportId/export?format=json|pdf|csv|spreadsheet|dashboard`: export an internal report.
- `GET /api/internal/iire/dashboard`: generate a dashboard-format report for the requested period.
- `GET /api/internal/iire/schedules`: list IIRE report generation schedules.
- `POST /api/internal/iire/schedules`: create daily, weekly, monthly, quarterly, or annual IIRE report schedules.
- `POST /api/internal/iire/run-due`: manually execute due IIRE report schedules.

IIRE reports include executive summaries, market intelligence, industry trends, revenue leakage trends, competitor movement, Competitor Relationship Graph intelligence, recommendation effectiveness, OSS distributions, behavioral analytics, churn intelligence, funnel analysis, Business DNA discoveries, outcome validation findings, intelligence accuracy metrics, Knowledge Growth Score, platform growth indicators, Intelligence Discovery Engine findings, Opportunity Discovery Engine findings, and anomaly alerts.

The IIRE worker starts automatically when `SYSTOLAB_IIRE_WORKER_ENABLED` is not `false`. Worker controls:

- `SYSTOLAB_IIRE_WORKER_ENABLED`
- `SYSTOLAB_IIRE_WORKER_INTERVAL_MS`

## Internal Platform Control Plane API

The Platform Control Plane is internal-only and requires:

```http
x-systolab-internal-key: your_internal_admin_key
```

Routes live under `/api/internal/platform` and expose module registry, job processing, analytics warehouse, AI analyst context, workspace intelligence, evidence repository, API governance, artifact versioning, disaster recovery, observability, data governance, intelligence validation, scan SLO, realtime refresh, governance contract, lineage, data quality, cost intelligence, graph intelligence, feature flags, and sandbox experiments.

The dashboard supports autonomous module discovery, operational stability scoring, scan replay intelligence, user journey intelligence, recommendation/decision intelligence, quality and accuracy views, security posture, infrastructure controls, graph/lineage exploration, feature flags, sandbox runs, and one-click PDF export.

## Authentication API

All authentication routes are self-contained and live under `/api/auth`.

- `POST /api/auth/google`: backend-verifies Google credential or sandbox `dev:` credential and creates a session.
- `POST /api/auth/otp/request`: creates a hashed, expiring OTP challenge with simulated backend delivery.
- `POST /api/auth/otp/verify`: verifies a single-use OTP challenge and creates a session.
- `POST /api/auth/password/register`: creates a password account and simulated signup OTP challenge.
- `POST /api/auth/password/login`: validates password credentials with 3-attempt lockout.
- `POST /api/auth/password/forgot`: creates a hashed reset token with simulated backend delivery.
- `POST /api/auth/password/reset`: verifies the reset token and updates the password.
- `POST /api/auth/refresh`: rotates refresh token and issues a new access token.
- `POST /api/auth/logout`: invalidates the active session.
- `GET /api/auth/me`: returns the bearer-token user and session.
- `GET /api/auth/sessions`: lists device sessions.
- `DELETE /api/auth/sessions/:sessionId`: revokes a device session.

Authentication uses one unified user record for Google ID, email, phone, password hash, providers, verification flags, retry counters, and lifecycle states. OTP and reset delivery is simulated by the backend; no external email, SMS, messaging, or OTP service is called.

## Public SYSTOLAB API

All `/v1` endpoints require:

```http
x-systolab-api-key: your_api_key
```

Development accepts `SYSTOLAB_DEV_API_KEY` from `.env`.

`POST /v1/scans`

Queues a scan through the SYSTOLAB scan worker and returns job metadata.

`GET /v1/snapshots/:snapshotId`

Retrieves the compact AICE Decision Object generated by the Decision Compression Layer.

`GET /v1/snapshots/:snapshotId/report`

Retrieves the full customer report `ReportSnapshot`.

## Snapshot Integrity

Full report snapshots contain:

- `integrity.snapshotHash`
- `integrity.evidenceHashChain`
- `integrity.immutableVerificationFingerprint`
- `executionProvenance`
- `rawSignalTelemetry`
- `validationTrace`
- `freshness`

These fields are retained for auditability in full reports. Snapshots are immutable in the Mongo model. Update operations are blocked for the `Snapshot` collection.
