# SYSTOLAB Intelligence Architecture

SYSTOLAB V1 uses a self-owned, deterministic intelligence stack. It does not call Google PageSpeed, paid SEO APIs, analytics APIs, ad APIs, AI scoring APIs, traffic APIs, CRM APIs, SMS providers, or external OTP services.

## Layer Flow

1. Identity/Context Layer
2. Data Layer
3. Truth/Evidence Layer
4. Intelligence Layer
5. Revenue Intelligence Layer
6. Confidence Layer
7. Automation Layer
8. Action/Alert Layer
9. Outcome Validation Layer

Layers communicate through `SystolabEventEnvelope` records. Each event includes the event type, layer, snapshot ID, workspace ID, target URL, timestamp, payload, evidence IDs, and confidence score. A layer writes its own artifacts and emits events; it does not mutate another layer's private logic.

## AICE Report Outputs

SYSTOLAB AICE uses one intelligence system with full report output plus a compact decision-object output.

The full `ReportSnapshot` contains crawler operations, evidence objects, validation traces, raw signal telemetry, system health, confidence factors, governance records, compatibility findings, proof layers, and operational learning data. It powers the customer report, PDF export, internal admin, debugging, system learning, and auditability.

The compact Customer Intelligence Plane is generated by the Decision Compression Layer in `decisionCompressionService.ts`. It remains available for API consumers that need only the locked `AiceDecisionObject`: risk level, time sensitivity, aggregated evidence summary, coverage score, confidence score, revenue impact range, if-not-fixed outcome, action window, final recommendation, access limitation, customer-safe impact statement, and heatmap summary.

Server credentials, auth tokens, environment secrets, database secrets, and private infrastructure secrets are never report fields and must not appear in any customer, API, PDF, or admin report output.

Evidence-bound conclusion rule: customer-facing risk, business impact, conversion effect, user intent, and value-unit ranges must be directly supported by current validated evidence. Incomplete, blocked, inaccessible, or insufficient evidence reduces confidence, may produce `UNKNOWN` risk, and must communicate limitations instead of extrapolated conclusions. Historical intelligence may guide acquisition strategy, but it cannot raise customer confidence when current-scan evidence is weak.

Current full-report routes:

- `GET /api/reports/:snapshotId`
- `GET /api/reports/:snapshotId/pdf`
- `GET /v1/snapshots/:snapshotId/report`

Current compact decision routes:

- `GET /api/reports/:snapshotId/decision`
- `GET /v1/snapshots/:snapshotId`

## V1 Active Engines

- Platform Control Plane provides module registry, distributed job metadata, warehouse rollups, governance controls, feature flags, sandbox experiments, API governance, observability, and internal operations endpoints.
- Scan History Tracking stores immutable target history through `ScanHistory`.
- Evidence Collection creates Evidence Objects and validation traces.
- OSS Scoring Engine produces deterministic score traces and visual states.
- Revenue Intelligence Engine estimates structural opportunity using OSS friction, conversion readiness, trust, information clarity, optional first-party lead volume, and competitor structural pressure.
- Recommendation Engine maps each issue to one executable action, one evidence set, one projected score movement, and one future validation plan.
- Evidence Database stores before/after evidence rows with confidence, EO references, recommendation IDs, validation trace IDs, and snapshot lineage.
- Recommendation Outcome Validation Loop maps recommendation -> action -> detected implementation -> measured improvement -> structural value-unit impact.
- Confidence Engine explains confidence through weighted factors, missing inputs, and evidence basis for every major estimate.
- Industry Benchmark Engine compares against SYSTOLAB-owned seeded vertical baselines for dentist, law firm, ecommerce, SaaS, and local service websites.
- Competitor Snapshot Analysis scans public competitor URLs and compares structural dimensions without rankings or market claims.
- Competitor Intelligence Engine creates competitor timelines and movement explanations from recurring scans.
- Competitor Relationship Graph records business-to-competitor relationships and aggregates market clusters, influence leaders, overlap, concentration, and growth velocity for internal IIRE/ODE use.
- Monitoring Scheduler creates daily, weekly, or monthly schedule records with next-run state.
- Monitoring Worker runs inside the API process, polls due schedules, launches scans, persists snapshots, advances next-run times, and records failures.
- Alert Engine generates dashboard alerts for score drops, competitor movement, validated recommendations, and revenue pressure.
- Notification Outbox stores dashboard and email-simulated delivery jobs without SMTP, SMS, or third-party messaging APIs.
- Operational Memory Graph emits graph-ready nodes and edges for website, snapshot, issue, recommendation, outcome, revenue impact, and competitor relationships.
- Business Evolution Engine tracks OSS timelines, trend, score delta, and cause narrative.
- Competitive Threat Radar classifies structural threat level from competitor gaps, competitor movement, and client regressions.
- Business DNA Engine identifies strengths, weaknesses, growth style, and recurring patterns.
- Edit Intelligence System records first-party dashboard funnel/edit events without third-party analytics.
- Lightweight Change Detection compares the latest prior snapshot for the same tenant and target against the current scan.

## Persisted Operational Memory

When MongoDB is connected, SYSTOLAB persists:

- `Workspace` for tenant, target, and business context.
- `ScanHistory` for chronological score and dimension movement.
- `IntelligenceEvent` for layer-to-layer event records.
- `RecommendationRecord` for action and automation readiness.
- `ChangeRecord` for before/after movement and recommendation validation.
- `EvidenceRecord` for before/after evidence database rows.
- `OutcomeValidationRecord` for recommendation outcome proof.
- `AlertRecord` for dashboard alert records.
- `MonitoringSchedule` for recurring scan cadence and next-run state.
- `NotificationOutbox` for self-owned alert delivery jobs.
- `EditEvent` for first-party dashboard behavior signals.
- `CompetitorRelationshipRecord` for graph-ready competitor relationship observations.
- `ModuleRegistryEntry`, `PlatformJob`, `AnalyticsWarehouseRecord`, `EvidenceArtifact`, `ApiGovernanceRecord`, `ArtifactVersionRecord`, `OperationalControlRecord`, `IntelligenceLineageRecord`, `GraphIntelligenceRecord`, and `FeatureFlagRecord` for the internal Platform Control Plane.
- `UserMemory` for graph-ready context records.

The in-memory fallback keeps local development functional when MongoDB is unavailable.

## History-Limited States

Several engines become stronger as scan history grows. On the first scan they return baseline or collector-ready states. After a second scan, SYSTOLAB can validate recommendation outcomes, detect score deltas, create competitor timelines, classify growth style, and generate stronger threat/DNA signals.

## Revenue Intelligence Rules

Revenue Intelligence V1 reports structural opportunity and value units only. It is not a currency forecast, conversion guarantee, ranking promise, traffic estimate from an external provider, or financial claim. Currency output requires first-party business inputs such as average order value, close rate, lead value, or CRM data.
